# Walkthrough - Fix Receiver Stuck at "Fetching Metadata"

## Real Root Cause

Since both sender and receiver run TorrentShare with the same 14 public trackers, the tracker-merging fix was not the core issue. The actual problem is **peer discovery for a brand-new torrent**:

- The sender creates a fresh torrent that **no public tracker has ever seen**
- The receiver adds the magnet and asks those public trackers "who has this hash?" — but the trackers have no answer yet (tracker announcements can take minutes to propagate)
- LSD (Local Service Discovery) works on the same LAN but fires on a fixed interval (~60s), so it's slow to start
- DHT also needs time to propagate knowledge of a new torrent

**Result**: the receiver can't find any peer to exchange metadata with, and sits at "Fetching Metadata..." indefinitely or for a very long time.

## Fix — Embed Sender's Direct IP + Port as `x.pe=` Peer Hint

The `x.pe=` magnet parameter is the BitTorrent standard for **direct peer address hints**. When the receiver parses a magnet link that contains `x.pe=192.168.1.10:6881`, libtorrent connects to that address immediately — no tracker, no DHT, no LSD needed.

### Changes

#### 1. `MagnetUtils.java` — Add `generateMagnetWithPeer()` overload
Add ability to include `x.pe=IP:port` in the generated magnet link.

#### 2. `TorrentManager.java` — Detect local LAN IP + session port, add `x.pe=` to seeding magnet  
In `startSeeding()`, discover the machine's LAN IP address and the jlibtorrent session's actual listening port, and embed them as a peer hint in the stored magnet URI.

#### 3. `MagnetUtils.java` — `extractPeerAddresses()` helper  
So `magnetWithAllTrackers()` can preserve `x.pe=` hints from the original magnet.

#### 4. `TorrentManager.magnetWithAllTrackers()` — Preserve `x.pe=` hints  
When the receiver processes the magnet, the `x.pe=` sender address must pass through unchanged.

## Result

When the receiver adds the magnet link, libtorrent immediately connects to the sender's LAN IP:port and requests the metadata. Peer exchange happens in <1 second on LAN, regardless of whether public trackers know about the torrent.

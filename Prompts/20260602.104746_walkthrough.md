## Summary of Fixes

### 1. `com/torrentshare/torrent/TorrentCreator.java`
* **Issue**: The `TorrentBuilder` class has no method named `.build()`.

### 2. `com/torrentshare/torrent/TorrentMonitor.java`
* **Issue 1**: The `.getTorrentHandles()` method returns a native JNI array `TorrentHandle[]` instead of `List<TorrentHandle>`.
* **Fix 1**: Replaced the type assignment from `List<TorrentHandle>` to `TorrentHandle[]` and iterated over the array directly (supported out-of-the-box by Java's enhanced for-loops).
* **Issue 2**: Deprecated `.isPaused()` and SWIG-specific enum structures (`state_t.checking`, `state_t.allocating`) caused lookup and type mismatch errors.
* **Fix 2**:
  * Implemented a robust bitwise AND check on the status flags via **`ts.flags().and_(TorrentFlags.PAUSED).non_zero()`** to verify if the torrent is paused.
  * Replaced SWIG enums with JLibTorrent's high-level **`TorrentStatus.State`** enum constants (`CHECKING_FILES`, `DOWNLOADING`, `SEEDING`, etc.).

### 3. `com/torrentshare/ui/TorrentDetailsDialog.java`
* **Issue**: Used the deprecated `.findTorrent(...)` method on `SessionManager` which does not exist in the wrapper.
* **Fix**: Aligned to use the high-level lookup method **`.find(new Sha1Hash(infoHash))`**.

### 4. `com/torrentshare/service/ShareService.java` & `com/torrentshare/ui/ShareTab.java`
* **Issue**: Called `ti.infoHash()` on JLibTorrent's `TorrentInfo` descriptor which was renamed in v2.
* **Fix**: Changed all instances of `ti.infoHash()` to **`ti.infoHashV1()`** to retrieve the correct SHA-1 hash for magnet generation and visual display.

### 5. `com/torrentshare/torrent/TorrentManager.java`
* **Issue 1**: Constructing `AddTorrentParams` and loading fastresume data was failing because `readResumeData(byte[])` was missing or mismatched.
* **Fix 1**: Rewrote `restoreTorrent` using the official JLibTorrent SWIG utilities:
  ```java
  com.frostwire.jlibtorrent.swig.error_code ec = new com.frostwire.jlibtorrent.swig.error_code();
  com.frostwire.jlibtorrent.swig.byte_vector buffer = com.frostwire.jlibtorrent.Vectors.bytes2byte_vector(resumeBytes);
  com.frostwire.jlibtorrent.swig.add_torrent_params nativeParams = com.frostwire.jlibtorrent.swig.add_torrent_params.read_resume_data(buffer, ec);
  AddTorrentParams atp = new AddTorrentParams(nativeParams);
  ```
  This cleanly reads bencoded fastresume structures.
* **Issue 2**: Bypassed non-existent `SessionManager.session()` calls and directly interacted with the underlying native SWIG JNI session object:
  ```java
  sessionManager.getJlibtorrentSession().swig().add_torrent(atp.swig(), ec);
  ```
* **Issue 3**: `AddTorrentParams.writeResumeData(params)` returns a JLibTorrent `Entry` instead of a raw `byte[]`.
* **Fix 3**: Converted the bencoded entry structure cleanly:

* **Issue 4**: Corrected `prepareShutdown()`'s handles iteration to process `TorrentHandle[]` instead of lists, cleanly flushing resume files.

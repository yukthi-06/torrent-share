# Implementation Plan - Fix File Sharing / Seeding

## Root Cause

In `TorrentManager.startSeeding()`, two bugs prevent file serving from working:

### Bug 1 — Wrong API: `download()` used instead of seeding
Line 132 calls `sessionManager.getJlibtorrentSession().download(ti, sourcePath.getParentFile())`.

`download()` tells jlibtorrent to **download** the torrent to the given folder — it does not know the files already exist locally. libtorrent will try to re-download the data instead of seeding it.

**Fix:** Use `AddTorrentParams` with `TorrentFlags.SEED_MODE` set. This tells libtorrent the data already exists at the given path and to begin seeding immediately without re-hashing.

### Bug 2 — Wrong save path: `sourcePath.getParentFile()`
The save path is set to `sourcePath.getParentFile()` — the parent directory of the source. This is wrong for both single files and directories:

- If the user drops a **file** `/data/docs/report.pdf` → save path = `/data/docs/` ✓ correct
- If the user drops a **directory** `/data/docs/` → save path = `/data/` ✗ wrong — it should be `/data/` only if the torrent name is `docs`, but the directory itself is what was hashed

**Fix:** The save path for seeding should be:
- For a **file**: `sourcePath.getParentFile()` (the directory containing the file)
- For a **directory**: `sourcePath.getParentFile()` (the parent of the directory, since the torrent name = the directory name)

So actually `sourcePath.getParentFile()` is correct in both cases for the save path — the issue is purely Bug 1.

## Proposed Changes

### [MODIFY] TorrentManager.java
Replace the `download()` call in `startSeeding()` with a proper `AddTorrentParams` + `SEED_MODE` approach.
